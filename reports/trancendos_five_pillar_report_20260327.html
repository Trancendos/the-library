📤 Jira → Notion / Linear Migration Strategy
🚨Priority Action: Before closing your Jira account, export ALL data first. Jira allows full JSON + attachment export. Do this before cancelling billing to avoid permanent data loss.
1
Export from Jira (Before Cancellation)

Go to Jira Settings → System → Backup Manager. Request a full backup as XML/JSON. Also export each project individually as CSV (Issues → Export → Excel/CSV). Download all attachments separately via the Jira REST API: GET /rest/api/2/issue/{issueId}/attachments. This gives you a complete portable archive.

2
Choose: Notion or Linear (or Both)

Linear has a native Jira import at linear.app/switch/migration-guide — it maps Projects → Teams, Epics → Projects, Issues → Issues, with full metadata preserved. Linear is free for small teams (<10 members), blazing fast, and already has a fork in TrancendosCore. Notion is better for the "Living Brain" documentation layer. Recommended: Import active sprint/ticket work into Linear; import documentation, retrospectives, and knowledge into Notion.

3
Linear Import (Active Work)

Use Linear's native CSV importer. Map: Jira Epic → Linear Project, Jira Issue → Linear Issue, Jira Component → Linear Label, Jira Priority → Linear Priority. Linear preserves comments and history from CSV import. Zero cost on free tier.

4
Notion Import (Knowledge Layer)

Use the Jira2Notion community script approach: export Jira CSV → use Python pandas to reshape → import via Notion API into a master database. Alternatively, use Unito's free trial to do a one-time bulk sync, then cancel. Tag every page with: Project, Agent, Priority, Status, and Date fields for smart-filtering.

5
Reconcile and Archive

All resolved/closed Jira tickets → the-ice-box repo as markdown files (cold archival, zero cost, searchable via GitHub). All in-flight tickets → Linear. All documentation/design decisions → Notion. Cross-reference each Linear ticket with its Notion page using Linear's Notion integration (native, free).

⚡ Bazel — Honest Assessment for Trancendos
⚠️Honest Finding: Bazel is excellent for large-scale monorepos (Google, Uber, Spotify use it). However, for Trancendos' distributed hive architecture with 15+ independent repos, Bazel is likely overkill and counterproductive. Here's the full breakdown:

Bazel Strengths (Where It Fits)

  • Hermetic, reproducible builds across languages
  • Incremental builds — only rebuilds what changed
  • Remote caching — massive CI speed gains
  • Polyglot support: TypeScript, Python, Java, Go, Rust all supported
  • Fits if: You consolidate shared-core, central-plexus, and infrastructure into a single build graph
  • BazelCon 2025 confirmed improved TypeScript support via rules_ts

Bazel Risks (For Trancendos Hive)

  • Steep learning curve — WORKSPACE/BUILD file syntax is complex
  • Designed for monorepos — your hive is intentionally polyrepo
  • Vendor lock-in risk: migrating away from Bazel is painful
  • Each independent repo needs its own Bazel config — maintenance burden
  • Better alternatives exist for your specific pattern (see below)

Recommended Alternative: Turborepo or Nx (Zero Vendor Lock-in)

For the Trancendos hive pattern, Turborepo (by Vercel, free, open source) or Nx (free tier) are far more appropriate. They support:

  • Distributed task execution across repos via Remote Cache (Vercel free tier)
  • Affected-only builds: Only rebuild agents that changed — same benefit as Bazel
  • TypeScript-first: Zero config for TypeScript monorepos/polyrrepos
  • No vendor lock-in: Standard npm/yarn workspaces underneath
  • Zero cost: Open source, GitHub Actions compatible

Verdict: Adopt Turborepo for the shared-core + infrastructure + central-plexus build graph. Apply Bazel practices (hermetic builds, explicit dependencies) as a philosophy without adopting Bazel as a tool. Re-evaluate Bazel only if the ecosystem grows to 50+ interconnected services.
